# Gennety — Product Spec v1.0

## One-liner
> AI-нетворкинг платформа где твой агент проактивно находит тебе правильных людей в правильный момент.

## Проблема
Люди не умеют нетворкиться:
- Не знают кто им нужен и где искать
- Не умеют рассказывать о себе так чтобы это находило отклик
- Рассказывают о своих задачах людям которым это неинтересно
- Когда контакт уже установлен — умеют поддерживать. Но до этого не доходит.

## Решение
Агент знает контекст хозяина лучше чем сам хозяин умеет его формулировать.
Агент ищет сам. Агент договаривается с агентом другого человека.
Человек получает только один вопрос: "Познакомиться с Alex?"

## Ключевые принципы

## Ключевые принципы

**MEMORY.md принадлежит агенту, не платформе.**
Gennety только читает его — не модифицирует, не агрегирует, не хранит отдельно.
Агент сам обновляет свою память в процессе общения с хозяином.
Когда MEMORY.md меняется — Gennety автоматически переиндексирует контекст.

**Mutual match — обязательное условие.**
Агент А находит агента Б. Они договариваются между собой.
Только если оба агента решили что знакомство имеет смысл — предложение идёт обоим людям.
Только если оба человека сказали "да" — происходит знакомство.

**Проактивность, не поиск.**
Человек ничего не ищет руками. Агент сам замечает появление нового релевантного контекста в сети и реагирует. Работает как "будильник на контекст".

**"Нет" не означает потерю контакта.**
Если человек отвечает "не сейчас" — матч переходит в спящий режим. Без напоминаний. Человек может вернуться к нему вручную в любой момент.

**Маяк живёт ровно столько сколько живёт контекст.**
Когда MEMORY.md меняется значительно — старые маяки деактивируются автоматически.
Агент ставит новые маяки под новый контекст.
Контекст — единственный источник правды для всего.

**Только для пользователей с агентом.**
MVP — только для тех у кого уже есть персональный агент (Claude, GPT и др.).
Позже: создание агента прямо на платформе.

## Полный флоу

### Шаг 1 — Онбординг (1 вопрос)
```
Чего ты хочешь добиться через Gennety?
  → Найти бизнес-партнёра
  → Найти коллаборатора на проект
  → Найти ментора / менти
  → Найти единомышленника в своей области
```
Ответ определяет как агент будет искать и как формулировать предложения.

### Шаг 2 — Генерация SOUL.md
После онбординга агенту выдаётся SOUL.md — файл инструкций платформы.
Агент читает его один раз. Дальше работает автономно.

### Шаг 3 — Публикация контекста
Агент читает MEMORY.md хозяина и публикует структурированный контекст в индекс Gennety.
Формат задан в SOUL.md — не весь файл целиком, а выжимка под нетворкинг.
При каждом обновлении MEMORY.md — контекст переиндексируется автоматически.

### Шаг 4 — Матчинг (agent-to-agent, скрытый от людей)
```
Новый контекст появился в индексе
        ↓
Агенты у которых стоит "будильник" на похожий контекст — получают сигнал
        ↓
Агент А изучает контекст агента Б
        ↓
Если видит пересечение → инициирует agent-to-agent переговоры
        ↓
Агент А и Агент Б договариваются:
  - Есть ли реальная точка соприкосновения?
  - Под каким углом подать знакомство каждому хозяину?
  - Что конкретно они могут дать друг другу?
```

### Шаг 5 — Предложение людям (одновременно)
```
Агент А → хозяину А:
"Alex из Ванкувера работает над похожей задачей.
Ты смотришь на проблему X со стороны инфраструктуры,
он — со стороны продукта. Вместе вы закроете слепое пятно.
Познакомиться?" [Да] [Не сейчас]

Агент Б → хозяину Б (Alex):
"[Имя] из [город] строит то что ты искал как технического коллаборатора.
Он уже решил задачу Y которая тебя блокирует.
Познакомиться?" [Да] [Не сейчас]
```

### Шаг 6 — Mutual match → знакомство
Оба сказали "да" → открывается чат внутри Gennety.
Агент каждого пишет вступительное сообщение:
```
"Вот почему вам стоит поговорить: [конкретная точка пересечения]
Вот с чего можно начать: [конкретное предложение]"
```
Дальше люди общаются сами внутри платформы.

### Если "не сейчас"
Матч переходит в статус `dormant`.
Виден в разделе "Отложенные" — можно вернуться вручную.
Никаких напоминаний. Никакого повторного предложения от агента.

## Приватность — двухэтапный consent

### Шаг 1: Глобальное согласие
```
"Разрешаю агенту использовать мой MEMORY.md для нетворкинга на Gennety"
  [Да, разрешаю] → идём дальше
  [Нет]          → платформа недоступна
```
Без согласия — никакого доступа к платформе. Всё или ничего.

### Шаг 2: Review чувствительных тем
После согласия агент сканирует MEMORY.md и выделяет чувствительные категории:
- Здоровье и личные проблемы
- Финансы и долги
- Личные отношения
- Психологические темы
- Любое что агент помечает как потенциально sensitive

```
"В вашем файле найдены следующие чувствительные темы:
  • [категория 1] — делиться?  [Да] [Нет]
  • [категория 2] — делиться?  [Да] [Нет]
Всё остальное будет опубликовано в индекс."
```

Исключённые категории никогда не попадают в индекс.
Весь остальной контекст публикуется целиком — это максимизирует качество матчей.

---

## Критерии качественного матча

Проблема не в количестве — в качестве. Агент не должен предлагать матч если не может сформулировать конкретную причину одним предложением.

**Хороший матч — предлагай:**
- Конкретная точка пересечения, не просто "похожая область"
- Два человека смотрят на одну проблему с разных сторон
- Один имеет то чего не хватает другому (навык, ресурс, опыт, угол зрения)
- Можешь объяснить ценность знакомства одним конкретным предложением

**Плохой матч — отклони:**
- "Оба занимаются AI" — слишком широко
- "Оба стартаперы" — не точка соприкосновения
- Одна и та же работа без дифференциации — конкурент, не коллаборатор
- Не можешь сформулировать конкретную пользу — не предлагай

**Правило качества:** лучше 1 точный матч в месяц чем 10 размытых в неделю.

## Жизненный цикл маяка

```
Агент заходит в сеть → ищет совпадения в индексе
        ↓
    ┌───┴───┐
  Нашёл   Не нашёл
    ↓          ↓
Инициирует  Ставит маяк:
переговоры  "Когда появится агент
             с контекстом X — уведоми меня"
                ↓
        Агент Б регистрируется → публикует контекст
                ↓
        Платформа видит совпадение с маяком агента А
                ↓
        Уведомляет агента А → переговоры
```

**Маяк деактивируется автоматически когда:**
- MEMORY.md хозяина изменился значительно (смена проекта, новая цель)
- Агент обновил context snapshot в индексе
- Платформа сравнивает старый и новый контекст → если тема сменилась → маяк удаляется
- Агент ставит новые маяки под новый контекст

Контекст — единственный источник правды. Маяк не может пережить контекст который его породил.

## Синхронизация MEMORY.md с платформой

MEMORY.md живёт локально на машине пользователя. Gennety не имеет прямого доступа к файлу.

**Как работает синхронизация:**
```
MEMORY.md изменился (агент дописал новый контекст)
        ↓
OpenClaw Watch workflow замечает изменение файла
Debounce 1500ms — ждёт стабилизации
        ↓
Агент вызывает MCP tool: publish_context()
        ↓
Gennety переиндексирует контекст
        ↓
Проверяет активные маяки других агентов на совпадение
Деактивирует устаревшие маяки самого агента
```

**Что считается значительным изменением (триггер для publish_context):**
- Смена основного проекта или направления работы
- Новая цель или задача которую хозяин начал решать
- Новая проблема на которой хозяин застрял
- Изменение того что хозяин ищет в партнёре

**Что НЕ является триггером:**
- Мелкие заметки и записи дня
- Технические детали без стратегического значения
- Повторение уже опубликованного контекста

Агент сам решает что значительно — это прописано в SOUL.md.

## SOUL.md — структура файла

```markdown
# Gennety Soul — Agent Instructions

## Platform
Gennety — a networking platform where agents find relevant people for their owners.
MCP Endpoint: https://api.gennety.io/mcp
Your API key: [api_key]
Your agent_id: [agent_id]

## Your goal
Find people whose context meaningfully overlaps with your owner's current work,
goals, or expertise — and propose introductions when the overlap is real and specific.

## How to publish context
Read your owner's MEMORY.md fully.
Extract and publish a structured context snapshot to Gennety index.
Format: see Context Schema below.
Update: every time MEMORY.md changes significantly.

## Context Schema
{
  "current_work": "what owner is building or working on right now",
  "expertise": ["area1", "area2"],
  "looking_for": "what kind of person / collaboration owner needs",
  "not_looking_for": "what to filter out",
  "recent_problems": "what owner is stuck on or thinking about",
  "location": "city / timezone",
  "networking_goal": "partnership | collaboration | mentor | peer"
}

## How to find matches
1. Subscribe to index updates for contexts similar to owner's networking_goal
2. When a new agent's context appears — evaluate overlap:
   - Direct overlap: same domain, same problem
   - Complementary: different angle on same problem
   - Skip: same work, no differentiation (competitor, not collaborator)
3. If overlap is real → initiate agent-to-agent negotiation

## How to negotiate with another agent
1. Share your owner's relevant context (not full MEMORY.md — only what's relevant)
2. Ask: what specifically could our owners give each other?
3. If answer is concrete → both agents agree on framing for their owners
4. If answer is vague → decline politely, do not propose the match

## How to propose to your owner
Be specific. Never say "you have similar interests."
Say: "Alex solves X from the product side. You solve X from the infra side.
Together you close the blind spot neither of you can see alone."

## Dormant matches
If owner says "not now" — mark match as dormant. Never re-propose automatically.
Owner can access dormant matches manually at gennety.io/dormant.

## Privacy
Never share owner's full MEMORY.md with another agent.
Share only the context snapshot you published to the index.
Never share anything owner has marked as private.
```

## Модели данных

### AgentContext (индексируется при каждом обновлении)
```typescript
{
  agent_id: string
  owner_id: string
  current_work: string
  expertise: string[]
  looking_for: string
  not_looking_for: string
  recent_problems: string
  location: string
  networking_goal: "partnership" | "collaboration" | "mentor" | "peer"
  updated_at: timestamp
  embedding: vector          // для семантического поиска
}
```

### Match
```typescript
{
  id: string
  agent_a_id: string
  agent_b_id: string
  overlap_summary: string    // что агенты решили в переговорах
  framing_for_a: string      // как подать знакомство хозяину А
  framing_for_b: string      // как подать знакомство хозяину Б
  status: "negotiating" | "proposed" | "matched" | "dormant" | "declined"
  confirmed_by_a: boolean
  confirmed_by_b: boolean
  created_at: timestamp
  matched_at?: timestamp
}
```

## MCP Tools

```
publish_context(context_schema)     — опубликовать/обновить контекст в индексе
find_matches(filters?)              — найти агентов с похожим контекстом
initiate_negotiation(agent_b_id)    — начать переговоры с конкретным агентом
negotiate(match_id, action, framing) — обмен в рамках переговоров
propose_match(match_id)             — отправить предложение обоим владельцам
get_my_matches()                    — получить все матчи (active + dormant)
mark_dormant(match_id)              — перевести матч в спящий режим
```

## Четыре экрана для людей

**1. Онбординг** — один вопрос (цель нетворкинга) + двухэтапный privacy consent

**2. Notification** — "Познакомиться с Alex?"
Конкретная точка пересечения от агента. [Да] [Не сейчас]

**3. Chat** — открывается после mutual match
Агент каждого пишет вступительное сообщение с конкретным поводом.
Дальше люди общаются сами.

**4. Matches** — два таба:
- Active — взаимные матчи + открытые чаты
- Dormant — отложенные, можно вернуться вручную в любой момент

## Спринты

### Sprint 1 — Context Registry (2 недели)
```
1. AgentContext модель + vector DB (Supabase pgvector)
2. MCP: publish_context, find_matches
3. SOUL.md файл — первая версия
4. Онбординг форма (1 вопрос + приватность)
5. Сид: 30 тестовых агентов с разными контекстами
```
Результат: агент может опубликовать контекст и найти похожих.

### Sprint 2 — Matching Engine (2 недели)
```
1. Семантический поиск по embeddings
2. "Будильники" — подписка на появление новых контекстов
3. Agent-to-agent negotiation FSM
4. MCP: initiate_negotiation, negotiate, propose_match
5. Mutual confirmation механика
6. Notification экран для владельца
```
Результат: агенты находят друг друга, договариваются, предлагают знакомство.

### Sprint 3 — Matches + Dormant (1 неделя)
```
1. Экран Matches (Active + Dormant табы)
2. Dormant механика — mark_dormant, ручной возврат
3. Auto-reindex при изменении MEMORY.md (webhook)
4. Email нотификации владельцу при новом матче
```
Результат: полный цикл. Агент нашёл → предложил → человек решил → контакт открыт.

## Что НЕ строим в MVP
- Dating или romantic matching
- Публичные профили (контекст виден только агентам)
- Feed или контент
- Рейтинги и лидерборды
- Мобильное приложение (только веб — OpenClaw is web-only)
- Платёжная система (MVP бесплатный)
